# Project Euler 147 Solution - Rectangles in Cross-hatched Grids

<https://projecteuler.net/problem=147>:

* [147.py](147.py)

The solver computes the cumulative number of rectangles in every cross-hatched grid up to `47 x 43`.

## Upright rectangles

For one `width x height` grid, the axis-aligned rectangles are counted by choosing two vertical and two horizontal grid lines:

```text
width * (width + 1) / 2 * height * (height + 1) / 2
```

The code sums this for each smaller grid.

## Slanted rectangles

The slanted rectangles are described in doubled coordinates along the two diagonal directions. For each half-step starting point, the older port walked every possible slanted width and then every possible slanted height until it hit the grid boundary.

The faster version keeps the same geometry but collapses the innermost walk. For a fixed start and slanted width `w`, the number of legal heights is

```text
min(previous height cap, x_limit - w)
```

where `x_limit` decreases by one as `w` grows. The other boundary only raises the raw limit, so the running cap is simply a capped descending arithmetic sequence. That means all heights for that start point can be added with one closed-form sum instead of a nested loop.

This keeps the implementation close to the original enumeration while removing the expensive per-rectangle simulation.

## Cumulative total

`solve(max_width, max_height)` loops through all grid sizes in the requested rectangle, caches symmetric slanted counts, and adds:

```text
upright_count(width, height) + slanted_count(width, height)
```

The checkpoints remain:

```text
R(3, 2) = 37
S(3, 2) = 72
```

## Complexity

The slanted count for a single grid now costs `O(width * height)` start-point work instead of walking every rectangle height. The target still evaluates all smaller grids, but the constant factor is much lower and the runtime stays comfortably below a second in Python.
