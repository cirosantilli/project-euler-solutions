# Addition-equation counting approach

## Main idea

We count valid equations without ever enumerating the integer values that appear in them.

An equation has the form:

- expression on the left,
- one `=` sign,
- expression on the right,
- each expression is a sum of positive integers with no leading zeroes.

The goal is to count all such strings of length at most `n`.

## Key technique 1: process the equation column by column

Instead of evaluating whole numbers, process the decimal expansion from right to left.

At a given digit column, each side only needs to know:

- how many numbers are still **active** at this column,
- how many of them **end** here,
- the current **signed difference carry** from lower columns.

This avoids storing the actual values of the numbers.

## Key technique 2: signed carry, not ordinary addition carry

If the digit sums in one column are `S_left` and `S_right`, and the carry from lower columns is `c`, then the next carry is

`c' = (c + S_left - S_right) / 10`

This carry can be **negative**.

That detail is essential. For example, equations like `10=1+9` only work if the state keeps a signed difference carry.

## Key technique 3: track only how many numbers continue

If `a` numbers are active on the left and `a'` continue to the next column, then `a-a'` numbers end here.
The same applies on the right.

There are binomial factors:

- `C(a, a')` ways to choose which left-side terms continue,
- `C(b, b')` ways to choose which right-side terms continue.

That accounts for the fact that terms are ordered in the string.

## Key technique 4: convert the digit condition into a bounded-sum count

For one column:

- continuing numbers contribute a digit in `[0, 9]`,
- ending numbers contribute a digit in `[1, 9]`.

After a small shift, the equality condition becomes counting solutions to a bounded sum with:

- some variables in `[0, 9]`,
- some variables in `[0, 8]`.

Those counts are precomputed once with short polynomial-style dynamic programming tables.

## Key technique 5: dynamic programming on total string length

The final DP state is:

- total characters used so far,
- active term count on the left,
- active term count on the right,
- signed carry.

Transitions consume exactly `active_left + active_right` new digit characters, one from every active number in the current column.

Initial states pay for:

- all plus signs,
- the single equality sign.

Whenever the state reaches `(0, 0, 0)`, it represents a complete valid equation.

## Complexity

Because `n = 50` is small, the state space is compact:

- at most 25 terms per side,
- carry bounded to a small interval,
- only 50 total characters.

That makes the DP fast enough in plain Python, with no external libraries.
